问题4.现在的脚本的keys是写死在lua里的，需要改成参数化，具体在D:\Github\dianping\src\main\java\com\hmdp\service\impl\VoucherOrderServiceImpl.java的223行
【问题4 傻瓜教程：Lua 传参到底怎么用（按你当前项目）】

先记住一条就行：
- Java 里 `execute(script, keys, args...)`
- Lua 里：
  - `KEYS[1..n]` 对应 Java 传入的 `keys` 列表
  - `ARGV[1..n]` 对应 Java 传入的 `args...`

你当前代码现状：
1) Java 调用（VoucherOrderServiceImpl）：
   - `stringRedisTemplate.execute(SECKILL_SCRIPT, Collections.emptyList(), voucherId, userId, orderId)`
   - 这里 `keys` 是空，所以 Lua 里只能用 `ARGV`
2) Lua 脚本（src/main/resources/seckill.lua）：
   - 用 `voucherId` 去拼 key：`seckill:stock:`、`seckill:order:`
   - `stream.orders` 也写死在脚本里

所谓“keys 参数化”就是：
- 不在 Lua 里拼/写死 key 名
- 改成由 Java 先把 key 算好，再传进 Lua 的 `KEYS`

标准改法（推荐）：

Java 侧（传 KEYS + ARGV）：
```java
String stockKey = "seckill:stock:" + voucherId;
String orderKey = "seckill:order:" + voucherId;
String streamKey = "stream.orders";

Long result = stringRedisTemplate.execute(
        SECKILL_SCRIPT,
        Arrays.asList(stockKey, orderKey, streamKey),
        userId.toString(), voucherId.toString(), String.valueOf(orderId));
```

Lua 侧（不再写死 key）：
```lua
local stockKey = KEYS[1]
local orderKey = KEYS[2]
local streamKey = KEYS[3]

local userId = ARGV[1]
local voucherId = ARGV[2]
local orderId = ARGV[3]

local stock = tonumber(redis.call('get', stockKey))
if stock == nil then
    return -1
end
if stock <= 0 then
    return 1
end
if redis.call('sismember', orderKey, userId) == 1 then
    return 2
end
redis.call('incrby', stockKey, -1)
redis.call('sadd', orderKey, userId)
redis.call('xadd', streamKey, '*', 'userId', userId, 'voucherId', voucherId, 'id', orderId)
return 0
```

你只要这样理解就不会乱：
- “和 Redis key 相关的参数”放 `KEYS`
- “业务值参数（userId、orderId 等）”放 `ARGV`

改完后的收益：
- key 不硬编码，脚本可复用
- 多环境/改前缀更容易
- 结构更清晰，排查问题更快

问题6.D:\Github\dianping\src\main\java\com\hmdp\listener\SeckillVoucherListener.java 中
目前秒杀订单的 MQ 消费采用默认 AUTO_ACK，
如果数据库写入或库存更新异常，会触发消息无限重试；
同时死信队列被当作正常队列处理，存在重复下单风险。

改进方案是：

正常队列消费失败时显式抛异常，让 MQ 接管重试；

死信队列消费前先根据 orderId 查数据库，已存在则直接丢弃；

保证订单表唯一键幂等，避免重复写入。

该方案不引入复杂事务，仅通过最小改动提高系统稳定性和可解释性。

问题9：前端保存商铺信息的时候，没有地址信息
最推荐的方案：地址 + 地图选点（Pin）
流程（最常见、最好用）

老板在前端填：省市区 + 详细地址（比如“XX路XX号XX大厦”）

前端接入地图控件（高德/百度/Google/Mapbox…）让老板搜索地址

地图上出现一个点，老板拖一拖/点一下确认（因为地址解析偶尔会偏）

提交给后端：

address（文本）

lng/lat（经纬度）

可选：adcode/citycode（行政区编码，方便筛选）

你后端怎么存

MySQL：存店铺基础信息 + x(lng), y(lat)（你代码里就是 x/y）

Redis：保存成功后 GEOADD shop:geo:<typeId> lng lat shopId

✅ 优点：精度高、体验好、老板也能理解
⚠️ 注意：要做“确认点”这一步，不要只靠自动解析
完整链路改造（无代码，后续按此补）：
1) 前端表单
   - 增加地址字段：省/市/区 + 详细地址
   - 增加地图选点控件：搜索地址 -> 自动定位 -> 人工确认点
   - 保存时提交：address + x(lng) + y(lat) + typeId (+ adcode/citycode)
2) 后端接口
   - POST /shop 保存：校验 address/x/y 不为空，成功后写 Redis GEO
   - PUT /shop 更新：比较旧 typeId/x/y，变更则移除旧 GEO 再写新 GEO
3) Redis GEO 同步
   - 新增：GEOADD shop:geo:<typeId> <x> <y> <shopId>
   - 更新：旧 key ZREM shop:geo:<oldTypeId> <shopId>，新 key 再 GEOADD
4) 数据一致性
   - DB 成功后再写 Redis，失败记录日志便于补偿
   - 如有批量店铺，可启动时从 tb_shop 重建 GEO
